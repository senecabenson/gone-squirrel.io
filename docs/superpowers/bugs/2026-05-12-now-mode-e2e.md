## Bugs

## Bug 1: `/api/focus/recommend` 500 — race on chunk creation (P2002 unique constraint)

**Severity:** major
**Found in:** L2 / L3 (every `/focus` cold-start that triggers recommend)
**Repro steps:**
1. Cold-load `/focus`
2. Pick energy + time (e.g. High + 30)
3. Observe first POST `/api/focus/recommend` returns 500; UI retries and second call returns 200.

**Expected:** Single POST returns 200 with recommendation, no chunk-creation race.
**Actual:** Prisma `P2002 Unique constraint failed on the fields: (taskId, chunkIndex)` — `prisma.taskChunk.create()` inside the recommend route tries to create chunks that already exist when concurrent/duplicate calls race.

**File(s) implicated:** `src/app/api/focus/recommend/route.ts:109-114`

```ts
chunkRows = await prisma.$transaction(
  sizes.map((min, i) =>
    prisma.taskChunk.create({
      data: { taskId: result.task.id, chunkIndex: i + 1, ... }
```

**Suggested fix:** Use `prisma.taskChunk.upsert` keyed on `(taskId, chunkIndex)` with composite unique, OR wrap the existence check + create in a serializable transaction, OR de-dupe the client call (React 18 dev double-effect likely re-invokes). Investigate whether the duplicate POST is also a client-side issue (StrictMode useEffect double-fire) — if so, fix the caller too.

**Side observation:** The route is being called twice in quick succession (likely React StrictMode dev double-invoke OR retry-on-error). Even with idempotent fix, look at why we POST twice.

## Bug 3: Scorer doesn't reward "fit quality" — always returns shortest-eligible task

**Severity:** major (core UX of Now Mode is the recommendation; this makes it predictable in a bad way)
**Found in:** L3
**Repro steps:**
1. Seed High tasks at 15/30/60/120 minutes (e2e1/e2e2/e2e3/e2e8).
2. POST `/api/focus/recommend` with `{ energy: "high", durationMin: 30 }` — expect e2e2 (30-min) or anything close.
3. Repeat for `durationMin: 60` and `durationMin: 90`.

**Expected:** Recommendation should prefer tasks whose chunk size is closest to the user's `durationMin`. A user with 60 free minutes should get the 60-min task, not the 15-min one.
**Actual:** All four ask durations (15/30/60/90) returned the same 15-min task (e2e1). Score components for the three tied tasks are identical on energy (1.0), deadline (0.2 default — no dueDate), variety (1.0); only `staleness` distinguishes them by a sub-thousandth, and the smallest-eligible task wins via stable-sort tie-break.

**File(s) implicated:**
- `src/lib/now-mode/score.ts:88-100` (scoreOne; no "fit" component)
- `src/lib/now-mode/score.ts:107-117` (scoreTasks; eligibility binary)

**Suggested fix:** Add a `fitScore` component, weighted ~0.15–0.20, computed from `|chunkDuration − availableMin| / availableMin` (smaller = better). Re-balance other weights so totals still sum to 1. Alternative: tighten eligibility — for `durationMin ≥ 30`, exclude tasks whose largest chunk is < 50% of `durationMin` so a 15-min task can't satisfy a 60-min ask. Cap that with a closest-match fallback when nothing fits.

## Bug 4: Reasoning string injects user `durationMin`, not chunk duration

**Severity:** minor (misleading UX)
**Found in:** L3
**Repro steps:** Same as Bug 3 — ask for 30 min, get a 15-min chunked task; reasoning reads `"...30 min of clean work."` even though the round will actually run 15 min.

**Expected:** Reasoning reflects what the user will actually do: `"…15 min of clean work."` (chunk duration) or "…knock out the first 15-min chunk of a 30-min task."
**Actual:** Template uses `ctx.durationMin` (user's available time) instead of `result.chunkDurationMin` (the round length).

**File(s) implicated:** `src/lib/now-mode/reasoning.ts` (template substitution), `src/app/api/focus/recommend/route.ts:121-128` (`pickReasoning` call site).

**Suggested fix:** Pass `chunkDurationMin` to `pickReasoning` and use that in templates that quote a duration. Keep `availableMin` only in templates that explicitly contrast it ("you have 60 min, this is just 15").

## Bug 5: Medium-energy ask always returns same "due tomorrow" task regardless of ask size

**Severity:** minor (correct given current weights; flag for tuning)
**Found in:** L3
**Repro:** POST `/api/focus/recommend` with `{ energy: "medium", durationMin: 15|30|45|60 }`.

**Actual:** All four return e2e9 ("Medium 15 due tomorrow") with the same 15-min chunk. The dueDate ≤ 24h boost (deadline = 1.0, weight 0.3 → +0.3 over peers) dominates and locks the result. **Suggested fix:** After completing the urgent short task, the round-complete screen could surface a "Got 45 min left — pick another for that window?" CTA. Out-of-scope for the scorer itself, but worth noting in the round-complete spec.

## Bug 8: Finish Later schedules chunks in DB but never pushes them to Google Calendar

**Severity:** blocker (the whole "see it on your calendar" promise of Now Mode silently fails)
**Found in:** L6
**Repro steps:**
1. Pick a task (any), start Pomodoro
2. Click `Done early` → `Finish later — pick something else`
3. Choose 45 min, click `Schedule it`
4. Observe toast `"Scheduled. 5/13/2026, 1:30:09 PM · 45 min"`.
5. Check DB: `TaskChunk.scheduledStart` populated ✓, `googleEventId` NULL.
6. Check Google Calendar: no event created.

**Expected:** After `POST /api/focus/finish-later` succeeds, a GCal event exists in the user's "GoneSquirrel Tasks" calendar with `extendedProperties.private.chunkId = <chunk.id>` and `extendedProperties.private.taskId = <task.id>`, and `TaskChunk.googleEventId` is set to the new event ID.
**Actual:** `SchedulingService.scheduleChunk()` only writes `scheduledStart`/`scheduledEnd` to the DB. There is no GCal push. The sync function `pushChunk()` exists in `src/services/google-task-sync.ts:374` but **is never imported or called anywhere**. `grep` confirms only `syncSingleTaskToGoogle` is called (from `/api/tasks/[id]/route.ts:302`), and that path operates on `Task`, not `TaskChunk`.

**File(s) implicated:**
- `src/services/scheduling/SchedulingService.ts:326-368` (scheduleChunk — should push to GCal)
- `src/app/api/focus/finish-later/route.ts:43-46` (caller — after scheduleChunk, no GCal trigger)
- `src/services/google-task-sync.ts:374` (`pushChunk` is the orphaned sync function)

**Suggested fix:** Call `pushChunk(userId, chunk)` inside `SchedulingService.scheduleChunk()` after the DB update, or in `finish-later/route.ts` after the for-loop. Mirror the pattern used by `syncSingleTaskToGoogle` in `/api/tasks/[id]/route.ts`. Same fix is needed wherever else chunks get scheduledStart set (the recommend route's chunk materialization path; on-demand re-chunk paths).

**Cascading impact:** L9 chunk completion (recolor sage green, delete remaining-chunk events) is also broken end-to-end because the events never existed in the first place. Whatever exists in `pushChunk` / `deleteChunkEvents` is dead code today.

## Bug 11: Sticky banner contains nested `<button>` elements (invalid HTML)

**Severity:** major (a11y + DOM correctness)
**Found in:** L7
**Repro:** Read `src/components/focus/StickyPomodoroBanner.tsx:45-80`. The outer wrapper is a `<button class="w-full">` that contains the Pause and Stop `<button>` children.

**Expected:** Buttons cannot be nested. The interactive root should be a `<div role="button" onClick>` or the Pause/Stop controls should sit alongside (not inside) the routing button.
**Actual:** `<button>...<button>Pause</button><button>Stop</button>...</button>` — browsers either hoist the inner buttons out (breaking layout) or silently disregard nesting; either way React's rendered DOM is invalid and event delegation depends on `e.stopPropagation()` to function.

**File(s) implicated:** `src/components/focus/StickyPomodoroBanner.tsx:45-80`.

**Suggested fix:** Convert the outer wrapper to `<div role="button" tabIndex={0} onClick={() => router.push('/focus')} onKeyDown={…}>`. Keep Pause/Stop as real `<button>` children. Remove the inner `e.stopPropagation()` plumbing in favor of plain handlers.
